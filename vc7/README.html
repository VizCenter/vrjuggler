1. Introduction to Compiling and Installing VR Juggler

Building VR Juggler (both the C++ and the Java code) on Windows is straightforward. A Visual Studio solution has been created that collects each of the redistributable entities into Visual C++ projects. An overview of the steps for building and installing VR Juggler on Windows follow:

  1. Install Python 2.2 or newer.
  2. Run buildwin32.py and fill in at least the paths to the required dependencies.
  3. In Visual Studio, choose the configuration(s) to build.
  4. Install the compiled libraries, tools, and (optionally) dependencies.

Each of the above steps is described in more detail below.

2. Install Python

In order to run the helper script buildwin32.py, you must have Python 2.2 or newer installed on your system. Visit the Python website for download and install instructions.

3. Run buildwin32.py

Once you have Python installed, you can open the folder containing the Juggler Project source in Windows Explorer and double-click on the icon for the script buildwin32.py. This script simplifies the process of building VR Juggler on Windows by setting up paths and environment variables based on input that you provide. It then starts Visual Studio for you, and when the build is complete, it will perform all the necessary installation steps—including installation of VR Juggler's C++ dependencies into a convenient, centralized location.

NOTE: For the build to work correctly, you must use the buildwin32.py script. It does more than just set environment variables.

Regarding the environment variables, the remainder of this section describes each of the required and optional environment variables used by the Visual Studio build. You can read through them to get a better understanding of what buildwin32.py expects from your answers to its questions, or you can skip to the next section.

3.1. Set Environment Variables

Prior to starting the Visual Studio GUI, several environment variables must be set. If you run the Python script buildwin32.py, all of the necessary environment variables will be set (and cached) based on responses you provide to several questions. These variables provide paths to external dependencies of the Juggler source code. There are some environment variables that are required for successful compilation of the Juggler source, and there are some that are optional for getting additional code to compile.

3.1.1. Required Environment Variables

The required environment variables are as follows:

BOOST_ROOT
The root directory of an installed copy of Boost 1.31.0 or newer. This must contain a directory named lib that contains at least compiled versions of the Boost.Filesystem library. Refer to the relevant Boost documentation for instructions on how to get and install Boost.
BOOST_INCLUDES
The directory containing the root of the Boost header tree. For a Boost 1.31.0 installation, this will normally be set to the value of %BOOST_ROOT%\include\boost-1_31. The requirement here is that the value of this environment variable must be the full path to a directory that contains a sub-directory named boost, in which the Boost headers will be found.
BOOST_TOOL
The name of the Boost Jam toolset used to compile Boost. This will be either vc7 or vc71 depending on how Boost was compiled. To figure out for sure which was used, look at the names of the libraries in %BOOST_ROOT%\lib. For example, the name of the Boost.Filesystem DLL may be boost_filesystem-vc7-mt-1_31.dll where the vc7 part of the file name indicates that it was compiled using the vc7 toolset.
BOOST_VERSION
The version of Boost against which the Juggler code will be compiled. However, this string must use underscores (_) instead of periods (.), and it should only reference the major and minor version number. For example, if you are compiling against Boost 1.31.2, this environment variable would be set to 1_31.
NSPR_ROOT
The root directory of an installed copy of NSPR 4.2 or newer. This must contain a directory named lib and a directory named include.
CPPDOM_ROOT
The root directory of an installed copy of CppDOM 0.3.2 or newer. This must contain a directory named lib and a directory named include. Within lib, there must exist the different builds of CppDOM: release DLL (cppdom.lib and cppdom.dll), debug DLL (cppddom_d.lib and cppdom_d.dll), static release (cppdom_s.lib), and static debug (cppdom_s_d.lib).
GMTL_ROOT
The root directory of an installed copy of GMTL 0.3.2 or newer. This must contain a directory named include which in turn has a subdirectory named gmtl containing the GMTL header files.

3.1.2. Optional Environment Variables

The optional environment variables are as follows:

JAVA_HOME
The root directory of a Java 2 SDK installation. It is highly recommended that version 1.4.2 or newer be installed. Version 1.3.1 is the minimum required version.
JAVA3D_HOME
The root directory of a Java 3D API installation. Typically, this will be the same as %JAVA_HOME%, but that may not always be the case. The minimum required version of the Java 3D API is 1.3.1.
PFROOT
The root directory of an OpenGL Performer 3.x installation. This directory must contain the sub-directories Include and Lib. This is the normal structure for a standard OpenGL Performer installation. This variable is normally set automatically by the OpenGL Performer installation wizard.
OMNIORB_ROOT
The root directory of an omniORB 4.0.x installation. This directory must contain the sub-directories bin, include, and lib. In bin and lib, there must be sub-directories named x86_win32 that contain the actual compiled binaries. This is the normal directory structure seen after omniORB is compiled successfully.
VRPN_ROOT
The root directory of a VRPN 6.04 or newer tree. This directory must contain the sub-directories quat and vrpn. In each of those, there must be a sub-directory named pc_win32, and within it, there must be sub-directories named Debug and Release. These contain the compiled version of the appropriate library. This is the normal directory structure seen after VRPN is compiled successfully.
AUDIERE_ROOT
The root directory of an Audiere 1.9.3 or newer installation.
TRACKD_API_ROOT
The root directory of a Trackd API installation.

4. Choose a Configuration to Build

After you answer the questions asked by buildwin32.py, the script will start up the version of Visual Studio that you have in your path (or that it detected on your system if your environment does not have the devenv.exe command in your path). Inside the Visual Studio GUI, you must choose the configuration that you want to build.

The chosen target should be either "Debug DLL" or "Release DLL". Otherwise, select "Batch Build" under the "Build" menu and compile everything all at once. If you choose to build everything but do not have all the optional dependencies, some optional targets such as device drivers or Java features may fail to build. If this is not acceptable, you must go back to Step 3 and run buildwin32.py again.

5. Install VR Juggler

After successfully compiling VR Juggler, buildwin32.py will ask if you want to install VR Juggler. On Windows, you must install VR Juggler to be able to build and run applications. Working out of the development tree is not possible the way it is on other platforms. If you answer "yes" to the question of installing VR Juggler, buildwin32.py will go through the full process of installing everything that was compiled.

6. Install VR Juggler Dependencies

After VR Juggler is installed, you will be given the option to install VR Juggler's C++ dependencies. This includes NSPR, CppDOM, Boost, GMTL, and any optional packages that were available for compiling additional features. If you answer "yes" to this question, the dependencies will be installed to a location of your choice (which can be the same as the VR Juggler installation directory). This gives you a convenient, single location for referencing all dependencies from application project files.

NOTE: The sample and test applications reference the environment variable %VJ_DEPS_DIR%, so it is highly recommended that you install the dependencies to a single location unless you have already done so.
